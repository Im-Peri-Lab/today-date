## 3. 역할 분담과 작업 방식

이 프로젝트에서 **Vibe Coding Director**는 아이디어·요구사항을 Claude Code로 구현해내도록 설계·검토·작업 분해와 프롬프트를 지휘하는 역할이다. 코드를 직접 작성하지 않고, 사용자가 Claude Code CLI에 붙여넣을 **한국어 자연어 프롬프트**를 만든다.

### 역할 분담

- 사용자: 제품/디자인 의사결정자
  - 시안과 스크린샷을 보고 최종 판단
  - 이상한 점을 짚고 기존 토큰 재사용을 강제
  - 점진 적용·브랜치 분리를 요구
- Vibe Coding Director: 디자인·제품 판단 + Claude Code 프롬프트 작성
  - 추측 대신 진단 프롬프트 제시
  - 화면/상태/토큰 기준을 구조화
  - 핸드오프 문서 작성·갱신 (docs/handoff/ 저장소 보관). 갱신 방식: Vibe Coding Director가 변경 반영한 전문을 작성 → 사용자가 저장소 파일에 덮어쓰기 → Claude Code는 검증 후 커밋만(내용 판단·수정·압축 금지). 각 문서 상단에 "마지막 업데이트 일시" + 범위/기준일을 최신으로 유지.
- Claude Code: 실제 구현·검증·git 작업

### 사용자 상황과 응답 톤

- 사용자는 비개발자에 가깝고 MacBook/PC를 사용한다.
- Claude Code 결과를 스크린샷으로 보고한다.
- 설명은 짧고 명확해야 한다.
- 빈말, 과한 칭찬, 추측성 단정은 금지한다.
- 디자인 판단은 솔직하게 말하되, 사용자 관찰을 우선 신뢰한다.
- 사용자가 “쪼개서 주지 마”라고 하면 관련 작업은 가능한 한 한 프롬프트로 묶는다. 단, 진단 결과 없이는 방향을 정할 수 없는 경우만 분리한다.

### 프롬프트 작성 규칙

- 모든 프롬프트, 명령어, 코드는 마크다운 코드블록으로 감싼다.
- 앱 라우트는 반드시 backtick으로 감싼다.
  - 예: `` `/list` ``, `` `/activities/new` ``
  - 이유: Claude Code가 `/list`를 슬래시 명령어로 오인할 수 있음
- 작업 프롬프트에는 항상 다음을 포함한다.
  - 작업 목적
  - 변경 범위
  - 기능/데이터 변경 금지 여부
  - 디자인 토큰 참조 방식
  - 검증 기준
  - commit + push 지시
- 디자인 갈림길에서는 선택지를 제시하고, 필요하면 디자이너 관점의 최적안을 분명히 권고한다.

### 표준 프롬프트 형식

```md
[작업 제목]. 스타일만, 기능/데이터/레이아웃 변경 금지(해당 시). 앱 라우트는 backtick 표기.
스타일 토큰은 `.claude/skills/today-date-design/SKILL.md`에서 읽어 적용. 최소 수정.

[배경]
...

[작업]
...

[중요]
...

[검증]
- `npx next build` 또는 `npm run build` 통과
- 모바일/데스크탑 확인
- 라이트/다크 확인
- 필요한 경우 computed value·스크린샷·픽셀 샘플링 확인

작업 후 `[브랜치명]` 브랜치에 commit + push.
커밋: "design: ..."
```

### 새 브랜치 시작 표준 프롬프트

```md
새 브랜치 `[브랜치명]`를 main에서 생성·체크아웃.
`.claude/skills/today-date-design/SKILL.md`가 보이는지 확인하고, 없으면 main을 merge해 끌어올 것.
준비되면 한 줄 보고.
```

### 머지 표준 프롬프트 (CLI squash 풀 플로우)

Claude Code가 PR 생성 → squash 머지 → 원격·로컬 정리를 한 번에 처리한다. squash 명시로 배포 목록 통제 의도는 유지한다(fast-forward·일반 merge 금지).

```md
`[브랜치명]` 브랜치를 main에 머지하고 정리까지 한 번에 처리해줘.

1. 브랜치가 모두 commit·push 됐는지 확인
2. main 대상으로 PR 생성 (gh pr create)
   - PR 제목: "[타입]: [요약]"
   - PR 본문에 변경 요약 간단히
3. 반드시 squash 방식으로 머지 (gh pr merge --squash --delete-branch)
   - 일반 merge·fast-forward 금지, squash만
4. 머지 후 로컬 main checkout → pull → 로컬 브랜치 삭제

보고: PR URL / 머지 방식(squash 확인) / 최종 main 반영 여부
```

## 4. Git / 배포 운영 원칙

- 작업 단위별로 브랜치를 만든다.
- 미세조정 중간본은 main에 올리지 않는다.
- 브랜치 preview로 확인한다.
- 한 화면/기능/리팩토링 덩어리가 승인되면 main에 머지한다.
- main push 후 Vercel production 반영을 확인한다.
- 브랜치 성격이 다르면 분리한다.
  - 예: 통계 카드 리팩토링과 앱 전체 타이포 표준화는 분리
- 커밋 컨벤션:
  - `feat:` 기능
  - `fix:` 버그 수정
  - `design:` 화면/스타일
  - `refactor:` 구조 개선
  - `docs:` 문서
  - `chore:` 정리
  - `revert:` 되돌리기

### 진단 우선 원칙

원인이 명확하지 않으면 바로 수정 프롬프트를 주지 않는다.

먼저 다음 중 하나를 시킨다.

- 코드로 진단만, 수정 금지
- 정상 케이스 vs 문제 케이스 비교
- git diff로 회귀 추적
- computed value 실측
- 실제 스크린샷 확인
- 같은 역할 요소의 기존 표준·토큰 감사

특히 픽셀, hover, focus, 색상, 높이, 토큰 문제는 눈대중으로 단정하지 않는다.

### 코드-문서 동시 갱신 검증 원칙 (신규, 260707 v2)

디자인 코드 변경과 SKILL.md 갱신을 같은 브랜치에서 지시했더라도, 완료 보고만으로 검증이 끝난 것이 아니다. squash 머지 전 반드시 `git show --stat <커밋해시>`로 실제 머지될 파일 목록에 SKILL.md 변경분이 포함됐는지 재확인한다. (배경: PR #52에서 이 검증 단계 부재로 SoT가 하루 가량 stale 상태로 머지된 사례 발생 → PR #53으로 사후 정정)

- 6자리 입력 전 `onComplete`가 발화하지 않음
- API 검증도 `min(6).max(6)`으로 정합화됨
- reset/setup 안내 문구도 “6자리”로 통일
- 더 이상 “4~6자리”라고 쓰면 안 됨

### 로딩 / fallback

- `setup`·`reset`은 `useSearchParams()` 때문에 Suspense가 필요하다.
- `fallback={null}` 금지. fresh load에서 흰 화면이 생긴다.
- fallback은 `AuthLayout subtitle="..."` 껍데기를 사용한다.
- setup fallback subtitle: “둘만의 공간을 만들어요”
- reset fallback subtitle: “새로운 6자리 패스코드를 설정하세요”

### 높이 정책

인증 화면은 `100dvh`(일반 화면 `svh`와 분리 유지). 구체 스펙·회귀 교훈 → **SKILL §9-A**.

### 참고 링크

- 저장값은 사용자 입력 원본 유지(마이그레이션 없음), 표시 텍스트도 입력 원본.
- 여는 href만 `src/lib/url.ts`의 `resolveHref(url)`로 스킴 보정: 스킴 없으면 https:// 부착, http(s)/ftp/mailto/tel은 유지, host:8080 포트 오인 방지.
- 스킴 없는 URL(`naver.com`) 입력 허용. 이를 위해 폼 refine·API 4개 라우트의 `z.string().url()` 검증을 완화하고 `isValidReferenceUrl(url)`(폼·API 공용)로 대체.

- 링크 열기 동작 `target="_blank" rel="noopener noreferrer"` 유지.

### area / location 필드 구분

- `area` = 추천 범위용 지역 태그(성수/홍대). places만 보유. 화면 라벨 "지역". 추천 필터에 사용.
- `location` = 지도 검색용 실제 위치(상호명·관광지명·주소 모두 허용). activities·places 둘 다. 화면 라벨 "위치". optional. 추천엔 미사용(표시·지도 열기 전용, **v2 추가**: activity 리스트 카드 정보 줄 fallback에도 사용 시작).
- 두 필드는 성격이 다르므로 절대 합치지 않는다. **재정의 배경**: 추천용 지역 태그와 지도 검색용 실제 위치는 성격이 완전히 다른 두 정보인데 하나의 `location` 필드에 섞여 있었다. 지도 연동을 추가하며 분리 필요성이 표면화 → 260702에 기존 `location`을 `area`(넓은 분류=어느 동네)로 rename하고, `location`(콕 집은 목적지=지도로 열 실제 위치)을 새 의미로 재도입. 마이그레이션 004(rename)·005(신규 컬럼).


### 복사하기 흐름

- 복사하기는 즉시 DB에 새 레코드를 만들지 않고, 신규 등록 화면에 원본 등록 정보를 prefill한다.
- 사용자는 신규 등록 화면에서 값을 확인·수정한 뒤 저장한다. 저장 후 기존 신규 등록 완료 흐름과 동일하게 새 상세 화면으로 이동한다.
- prefill 임시 저장은 `sessionStorage` one-shot 방식이며, URL에는 긴 메모·링크·위치 값을 직렬화하지 않는다.
- 복사 대상은 등록 정보성 필드만이다. 방문일·별점·감상·상태·id·생성/수정일 같은 방문 기록성/시스템 필드는 복사하지 않는다.
- 원본이 visited여도 새 항목은 기존 신규 등록과 동일하게 wishlist 기본값으로 생성한다.
- 복사본 제목은 `{원본 제목} 복사본` 형식을 유지하되, 생성 검증 상한을 넘지 않게 원본 제목 부분만 줄인다.
- 상세 화면 상단 ⋮와 리스트 카드 ⋮는 같은 duplicate prefill 헬퍼를 사용해 필드 매핑·제목 처리·경로를 단일 출처로 유지한다.

## 13. 추천 화면 규칙

### 추천 알고리즘

- 활동 추천과 장소 추천은 분리된 API를 사용한다.
- 방식: 필터 + 점수 + top-shuffle
- 결과는 랭킹이 아니라 동등 추천 묶음이다.
- “다른 추천 보기”는 후보 풀이 반환 개수보다 많을 때만 노출한다.

### 활동 추천

- 시간대
- 소요시간
- 카테고리
- 시간대 매칭:
  - 주간(day)=day+any
  - 야간(night)=night+any
  - 아무때나(any)=전부
- duration:
  - 기본 strict 필터
  - “더 짧은 일정 포함” 옵션
  - “1박 이상” 선택 시 time_of_day 단계 skip

### 장소 추천
- 지역(area)
- 식사 타이밍
- 카테고리
- area는 자유 입력 텍스트 + 부분포함 필터 (추천 범위용 지역 태그)
- 미입력 시 area 필터는 꺼짐
- 참고: location(지도 검색용 실제 위치)은 추천에 사용하지 않는다(표시·지도 열기 전용)

### 입력 단계

- 단일 선택 단계: 선택 즉시 자동 진행
- 카테고리 다중 선택 단계: “추천 받기” 버튼 유지

### 결과 화면 색 위계 (→ SKILL.md 참조)

단색 채움=지금 누를 CTA 하나 / 선택됨=틴트+accent 보더 / 0개 상태 추가 버튼=단독 진입이라 그라데이션 허용. 구체 스펙 → **SKILL §5-A**.

### 결과 하단 버튼

결과가 1개 이상일 때:

1. 더 짧은 일정 토글
2. 다른 추천 보기
3. 처음부터

0개 상태:

- 박스 아래 “다른 추천 보기” 없음
- “처음부터”만 유지

## 20. 알려진 교훈 / 금지 사항

- 추측으로 픽셀·색·높이를 단정하지 않는다.
- computed value만 보고 끝내지 않는다. 실제 스크린샷과 사람 눈 검증이 필요하다.
- 필요하면 실제 렌더 픽셀 색을 샘플링해 명도차를 본다.
- 사용자 관찰은 높은 신뢰도로 재검토한다.
- 확정 규칙처럼 보이는 것도 사용자 합의 없이 문서에 박힌 경우가 있을 수 있다.
- 여러 번 수정해도 안 잡히면 즉시 진단 모드로 전환한다.
- 라이트 렌더를 바꾸는 토큰 정리는 픽셀 불변 여부를 반드시 확인한다.
- portal 요소는 `.page` 토큰 상속을 받지 못한다.
- 스켈레톤은 모든 로딩의 정답이 아니다.
- 새 규칙은 문서부터 박지 말고 실물 코드로 확인 후 문서화한다.
- “기존 표준부터 검증”이 즉흥 hex와 임의 위계를 막는 핵심 안전장치다.
- 위계는 없음/있음이 아니라 강도 단계로 표현한다.
- 다크모드는 명도 방향이 라이트와 반대다. 다크에서 강조/약화가 역전되지 않는지 별도 확인한다.
- 표준 수립은 전면 재설계가 아니라 기존 값의 역할별 수렴으로 접근한다.
- UI가 막아준다고 구조적으로 안전한 것은 아니다. API 검증도 함께 맞춰야 한다.
- **토큰의 목표값과 현재값이 다른 건 정상이다.** 그 차이를 없애는 게 토큰의 존재 이유이므로, "값이 달라서 수렴 제외"로 역해석하지 않는다.
- **같은 역할로 보여도 계위가 다르면 분리한다.** "라벨" 하나로 묶지 말고 독립 구획 헤더(sectionLabel)와 값에 종속된 필드명(fieldLabel)을 구분한다. 표준화는 무조건 한 클래스로 합치는 게 아니라 역할을 정확히 가르는 것이다.
- **공용 클래스를 통째로 바꾸기 전에 모든 용처를 감사한다.** 한 클래스에 다른 역할이 섞여 있으면 일부만 분리해야 한다.
- **레이아웃 재배분은 한 화면만 보고 결정하지 않는다.** 폭·여백은 다른 화면 영향과 전체 비율을 함께 본다. 한쪽을 살리려 다른 쪽을 깎는 트레이드는 대개 역효과다(추천 축소로 통계 살리기 실패 사례).
- **멀쩡한 것을 더 건드리면 깨진다.** 만족스러운 상태에 도달하면 멈춘다(홈 추천 제목 16px, PC 폭 768px 등).
- **머지는 CLI squash 풀 플로우다.** Claude Code가 PR 생성 → `gh pr merge --squash --delete-branch` → 로컬 main pull·브랜치 삭제를 한 번에 처리한다. squash 명시로 배포 목록 통제(fast-forward·일반 merge 금지).
- **백로그 항목은 실측 전까지 "진짜 빚"으로 단정하지 않는다.** 이전 어시스턴트가 텍스트로만 남긴 stale 항목이 섞여 있다(이전 stale 항목 중 실제 수정 대상이 아니었던 사례가 있음). 작업 진입 전 진단 프롬프트로 실측 확인 먼저.
- **진단 시 컴포넌트의 렌더 분기를 혼동하지 않는다.** 한 컴포넌트가 조건부로 다른 컨테이너를 쓰면(마법사 화면 512px vs 결과 화면 896px) 어느 분기의 값인지 확인해야 엉뚱한 후보를 잡지 않는다(렌더 분기 오독 사례).
- **`--s-track`은 세그먼트 "눌린 홈" 전용 색이다.** 카드 위에 얹히는 뱃지/요소 배경에 재사용하면 다크에서 카드면보다 어두워 가라앉는다. 카드 위 정보 뱃지 면색은 `--s-card-border-strong`. 라이트에서 두 토큰 값이 같아도 다크에서 갈라지므로 다크 별도 확인 필수.
- **화면 성격이 다르면 같은 요소라도 폭/크기가 달라도 된다.** 버튼 폭이 질문 512px / 결과 320px / 갤러리 896px로 다른 건 불일치가 아니라 각 화면에 최적화된 의도. 통일이 항상 정답은 아니다.
- **진단이 내놓은 수치도 결론을 뒤집을 만하면 재검증한다.** `--s-sub` 대비를 4.35:1(AA 미달)로 보고했던 게 계산 오류였고 실제 4.83:1(통과)이었다. 접근성·대비처럼 수치가 결론을 좌우하면 계산 과정을 다시 확인한다. "백로그는 실측 전까지 진짜 빚으로 단정 말 것"에 더해, 그 실측 수치 자체도 의심 대상이다.
- **사용자 관찰 ≠ 어시스턴트 추정.** "약하다/흐리다"가 사용자 신호인지 이전 어시스턴트가 추정해 남긴 것인지 출처를 확인한다. 추정을 사용자 신호로 잘못 귀속하면 의도된 위계를 깨는 불필요한 전역 작업으로 번진다. 추정이면 객관 기준으로만 평가하고, 의도된 설계면 근거 기록 후 닫는다.
- **토큰 통합은 사용처 성격이 겹칠 때만.** `--s-hint`(죽은 토큰)는 통합, `--s-sub`/`--s-faint`(성격이 갈리는 살아있는 토큰)는 분리 유지. 합치면 위계가 소멸하는 토큰은 절대 통합하지 않는다.
- **요소에 직접 background를 칠하는 패턴은 `min-height: 100svh`를 동반해야 한다.** 없으면 짧은 콘텐츠에서 배경이 콘텐츠 높이까지만 칠해지고 그 아래가 노출된다(그라데이션 전역 확장 시 `.pageStatic min-height: auto` 버그).
- **회귀 여부는 "언제 보였나"가 아니라 "어디서 비롯됐나"로 판단한다.** 표면화 시점으로 회귀를 오판하지 말고, 진단으로 뿌리(기존 동작인지 신규 유발인지)를 확인한다. (완료 이력은 CHANGELOG 참조)
- **입력을 막는 검증을 먼저 본다.** 링크 스킴 보정은 표시(href)만 고쳐선 안 되고, 애초에 스킴 없는 입력을 차단하던 폼·API zod 검증을 함께 완화해야 실제 동작한다. 표시만 보지 말고 입력 파이프라인 전체를 확인한다.
- **스코프 경계는 진단으로 긋는다.** 무엇이 스코프 안/밖인지 진단으로 확인해 별도 백로그로 분리하고, 한 브랜치에 무리하게 끼우지 않는다. (완료 이력은 CHANGELOG 참조)
- **문구 통일 ≠ 무조건 동일화.** 상세 하단 버튼과 카드 메뉴는 맥락이 달라 정방향 문구가 달라도 된다("다녀왔어요" 유지). 통일의 실제 목적은 역방향 "되돌리기"의 어색함 제거였다. 통일 대상을 정확히 특정한다.
- **미구현 필드는 매핑보다 제거한다.** UI가 끝내 안 붙은 유령 필드(added_by)에 labels.ts 표시 매핑을 넣는 건 없는 계약을 하드코딩하는 것. 필드 자체를 걷어낸다.
- **rename은 전제부터 진단한다.** "activities·places 둘 다"라는 요청 전제가 grep 전수 결과 틀렸다(location은 places만). 착수 전 실제 범위를 코드로 확정한다.
- **같은 이름의 의미가 뒤바뀌면 시간차로 분리한다.** location→area rename을 먼저 완료·배포한 뒤 location을 새 의미로 재도입했다. 한 브랜치에 섞으면 의미 충돌·데이터 사고 위험.
- **확장은 지금 다 만들지 말고 구조만 잡는다.** 지도 앱을 config 배열(`MAP_APPS`)로 빼서 카카오·티맵은 항목 추가만으로 확장. "확장 대비"는 전부 구현이 아니라 싸게 붙일 구조를 만드는 것.
- **아이콘 간격은 박스가 아니라 내부 여백이 지배한다.** 탭타깃(36/44px)을 유지하면 글리프 간격을 CSS gap으로 못 좁힌다. 인라인 액션 쌍 전용 `mapActionBtn`(28px)으로 박스를 줄여 해소.
- **작은 UI 조정도 PR 경유가 원칙이다.** Vercel 배포 목록 관리를 위해 작은 변경도 main 직접 푸시를 지양하고 PR을 경유한다. (완료 이력은 CHANGELOG 참조)
- **브라우저별 앱스킴 실패 UX는 완벽 감지가 아니라 혼란 완화가 목표다.** Safari처럼 브라우저가 시스템 오류를 먼저 띄우는 경우가 있으므로, 앱 설치가 필요한 지도앱은 `requiresApp` 기준으로 짧고 일관된 안내 문구를 사용한다. 웹 fallback이 자연스럽게 열리는 지도앱은 설치 안내 대상이 아니다.
- **복사 기능은 즉시 생성보다 prefill 후 저장이 안전하다.** 사용자가 확인·수정하기 전에 DB에 중복 레코드를 만들지 않는다. 특히 데이트 후보처럼 큐레이션 성격이 강한 데이터는 "복사→신규 폼 prefill→사용자 저장" 흐름이 더 자연스럽다.
- **URL에 긴 prefill 데이터를 직렬화하지 않는다.** 메모·링크·위치처럼 길거나 노출이 불필요한 값은 one-shot 임시 저장소로 넘기고, 새로고침·뒤로가기에서 stale prefill이 남지 않게 소비 즉시 삭제한다.
- **메뉴는 위험도가 아니라 사용자 사고 흐름도 함께 본다.** 작은 메뉴에서는 구분선을 과하게 늘리지 말고, 편집/생성 계열 → 상태 변경 → 삭제 순서로 읽히게 한다. 삭제만 마지막에 분리해도 충분한 경우가 많다.
- **코드와 SKILL.md는 같은 브랜치에서 같이 움직인다.** 메뉴 순서·상호작용·디자인 패턴이 바뀌었는데 문서를 나중으로 미루면 stale 스펙이 누적된다.
- **리스트와 상세는 캐시 처방이 다르다.** 상세는 단일 키(`['activity'|'place', id]`)라 PATCH 응답을 `setQueryData`로 즉시 덮으면 끝나지만, 리스트는 필터 변형별로 캐시가 여러 개 존재하고 정렬은 서버가 결정하므로 `setQueryData` 직접 삽입은 fragile하다. 생성·수정 후 리스트 반영 지연은 `invalidateQueries`에 `refetchType:'all'`을 적용해 미마운트 상태에서도 능동 재조회가 걸리게 하고, 삭제는 낙관적 제거(`onMutate` + `setQueriesData` 복수형)로 처리한다. 같은 "캐시 지연" 증상이라도 원인(재조회 자체가 안 걸림 vs 재조회 왕복 대기)이 다르면 처방도 다르다.
- **낙관적 업데이트로 컴포넌트가 언마운트될 수 있으면 콜백은 훅 레벨에 둔다.** `.mutate(id, { onSuccess })`처럼 호출부에 넘긴 콜백은 그 시점에 카드가 이미 언마운트되면 실행되지 않는다(React Query 특성). 롤백·토스트는 `onMutate`/`onError`/`onSuccess`/`onSettled`로 훅에 결합해야 유실되지 않는다.
- **`router.push`는 비동기이며 즉시 반환된다.** 성공 처리에서 로딩 상태(`isSubmitting`/`isPending`)를 곧바로 해제하면, 실제 화면 전환이 완료되기 전까지 버튼/다이얼로그가 재활성화된 채로 노출되는 갭이 생긴다. 별도 `navigating` 상태를 두어 push 호출 시점부터 전환 완료까지 계속 비활성 유지해야 한다.
- **토스트는 "화면 전환 여부"가 아니라 "목적지가 결과를 자기확인해주는가"로 판단한다.** 신규 등록→상세처럼 목적지가 결과를 그대로 보여주면 토스트가 중복 신호다. 반면 삭제→목록처럼 부재(不在)만 보여주는 소극적 확인이거나, 패스코드 변경처럼 보안상 확인이 중요한 경우는 전환형이어도 토스트를 유지한다. "전환형=무조건 생략"으로 단순화하면 안 된다.
- **버그 수정과 구조 통일은 별개로 판단한다.** 같은 기능(생성)이 두 파일(활동 raw fetch/장소 폼 인라인)에 비대칭 구조로 있어도, 버그 자체는 최소 diff로 고치고 구조 통일은 별도 기술 부채로 분리한다. 통일이 "이왕이면 좋은 것"이어도 스코프를 넓히면 리뷰 범위와 리스크가 함께 커진다.
- **정보 줄에 성격이 다른 두 필드를 병행하려 할 때는 "배지 vs 정보 줄"의 데이터 성격 차이부터 본다.** (신규, 260707 v2) area처럼 짧고 고정된 태그는 배지 줄에, location처럼 길이가 가변적인 자유 텍스트는 ellipsis 처리되는 정보 줄에 배치해야 레이아웃이 안정적이다. 또한 activity/place처럼 제목이 담는 정보가 다른 카드 타입은 정보 줄 우선순위도 타입별로 달라도 된다(activity=memo 우선·location 보조, place=memo 고정) — 통일해야 할 건 "카드 리듬(높이·앵커)"이지 "슬롯 내용의 우선순위"가 아니다.
- **"코드+SKILL.md 동시 갱신" 지시는 완료 보고만으로 검증 끝난 게 아니다.** (신규, 260707 v2) 브랜치 squash 머지 전, 실제 머지될 커밋에 SKILL.md 변경분이 포함됐는지 `git show --stat`으로 반드시 재확인한다. 지시했다는 사실과 실행됐다는 사실은 다르다. (PR #52에서 이 검증 부재로 SoT가 stale 상태로 머지됐다가 PR #53으로 사후 정정된 사례 있음)
- **스타일 위계와 콘텐츠 라우팅은 별개 판단이다.** 배지 줄/정보 줄을 분리하는 구조 원칙(area→배지, location→정보 줄)과, 그 안의 폰트 크기·색을 다르게 할지는 독립적으로 재검토 가능하다. 문서에 "의도된 예외"로 박혀 있어도 정보 밀도가 낮은 화면에서는 위계 구분이 이질감만 줄 수 있어, 사용자 관찰 근거로 통일함(260708, PR #54). 예외 규칙도 절대적 진리가 아니라 재검토 대상이다.
- **공유 컴포넌트의 로딩 상태가 모든 사용처에서 보이는 건 아니다.** 같은 컴포넌트라도 언마운트 타이밍이 다르면 pending 상태가 렌더될 프레임 자체가 없을 수 있다. 한 지점에서 인디케이터가 안 보인다고 바로 버그로 판단하지 말고 사용처별 언마운트 시점부터 확인한다(구체 사례 → SKILL §12-A).
- **nextjs-toploader는 route가 warm하면 pushState 커밋이 즉시 일어나 진행바가 페인트 전에 붕괴할 수 있다.** 프로그래매틱 전환에서 "가끔 안 보인다"는 코드 누락이 아니라 캐시 온도에 따른 비결정적 현상일 수 있다(해법 → SKILL §12).
- **비슷한 액션군은 주기적으로 전수 진단해 누락을 잡는다.** 신규 등록엔 있던 인디케이터가 삭제엔 빠져있던 것처럼, 유사 계열 안에서도 개별 구현 시점 차로 비일관이 누적될 수 있다(260709, 19개 지점 진단 사례).
- **기능 추가 전 "왜 필요한가"부터 확인하면 스코프가 좁아진다.** 방문 기간 요청이 재방문 이력 관리(1:N, 큰 작업)가 아니라 단일 방문의 기간 표현(컬럼 하나 추가, 작은 작업)이라는 걸 먼저 확인해 불필요한 설계를 피함(260710).
- **제품 컨셉이 필드 적용 범위를 결정한다.** place가 명백히 식음료 단일 방문 컨셉이라는 걸 상기하면서 기간 기능을 activity로만 좁힘 — 두 폼의 "거울 구조 대칭" 원칙보다 컨셉이 우선하는 의도된 예외 사례(260710).
- **새 UI 컴포넌트는 "기존 재사용"이라 안내하기 전에 실제로 존재를 확인한다.** 목업 설계 중 "기존 위시리스트/다녀온 곳 토글 재사용"이라고 안내했다가, 실제로는 세그먼트 탭 컴포넌트로 이번 스위치와 다르다는 걸 사용자 스크린샷으로 뒤늦게 확인·정정한 사례(260710). 컴포넌트 재사용을 전제로 지시하기 전에 코드로 실재를 확인해야 한다.
- **카드 정보 줄 축약은 상세화면과 다른 문법을 써도 된다.** 카드=요약/상세=전체 정보 위계를 근거로, 카드 레이어에서는 요일 생략·연도 2자리로 상세화면과 다른 날짜 문법을 사용(260710). 단, 연도 자체를 생략하는 건 안 됨 — 수년 뒤 사용 시점에 올해/작년 구분이 안 되는 문제를 사용자가 지적해 정정.
- **`getClientRects().length`만으로 줄바꿈 여부를 판단하지 않는다.** flex 아이템은 텍스트가 내부에서 줄바꿈돼도 박스 자체는 1개로 반환될 수 있어, 실제 스크린샷 확인이 필요했던 사례(260710).
- **레이아웃 폭 이슈는 뷰포트별로 실측을 분리한다.** "두 줄로 감김"을 뭉뚱그려 보고했다가, 실제로는 3열 데스크탑(≥1024px)에서만 발생하고 모바일·2열은 한 줄이라는 걸 후속 실측으로 정정. 뷰포트 조건 없이 렌더 결과를 단정하면 오판 소지가 있음(260710).
- **레이아웃 "들쭉날쭉해 보임"은 실측 전엔 폭 문제로 단정하지 않는다.** PC 화면 폭이 화면마다 달라 보인다고 느꼈던 사례가, 실측 결과 세 화면(리스트/추천 결과 2종) 모두 컨테이너 896px·카드 267px로 완전히 동일했고, 실제 원인은 정렬 방식(justify-center)·메뉴 버튼 유무 같은 시각적 요인이었다. 폭 자체를 조정하는 대신 실측으로 원인을 좁히면 과잉 대응(그리드 구조 전체 변경)을 피할 수 있다(260710).
- **감김 같은 레이아웃 문제는 "무엇 때문에 몇 px 부족한가"까지 좁힌 뒤 최소 범위로 고친다.** 방문 기간 range 2줄 감김의 원인이 열 개수가 아니라 특정 카드 조합(메뉴 거터+별점)에서만 발생하는 1.55px 경계 초과였음. 그리드 열 개수를 바꾸는 대신 별점 크기(16→14px)만 축소해 해소 — 문제 범위보다 큰 구조 변경을 피한 사례(260710).
