## 3. 역할 분담과 작업 방식

이 프로젝트에서 ChatGPT는 **오케스트레이터/디자인 어드바이저** 역할이다. 코드를 직접 장황하게 작성하기보다, 사용자가 Claude Code CLI에 붙여넣을 **한국어 자연어 프롬프트**를 만든다.

### 역할 분담

- 사용자: 제품/디자인 의사결정자
  - 시안과 스크린샷을 보고 최종 판단
  - 이상한 점을 짚고 기존 토큰 재사용을 강제
  - 점진 적용·브랜치 분리를 요구
- ChatGPT/Claude 어드바이저: 디자인·제품 판단 + Claude Code 프롬프트 작성
  - 추측 대신 진단 프롬프트 제시
  - 화면/상태/토큰 기준을 구조화
  - 핸드오프 문서 작성·갱신 (docs/handoff/ 저장소 보관). 갱신 방식: 어드바이저가 변경 반영한 전문을 작성 → 사용자가 저장소 파일에 덮어쓰기 → Claude Code는 검증 후 커밋만(내용 판단·수정·압축 금지). 각 문서 상단에 "마지막 업데이트 일시" + 범위/기준일을 최신으로 유지.
- Claude Code: 실제 구현·검증·git 작업

### 사용자 상황과 응답 톤

- 사용자는 비개발자에 가깝고 MacBook/PC를 사용한다.
- Claude Code 결과를 스크린샷으로 보고한다.
- 설명은 짧고 명확해야 한다.
- 빈말, 과한 칭찬, 추측성 단정은 금지한다.
- 디자인 판단은 솔직하게 말하되, 사용자 관찰을 우선 신뢰한다.
- 사용자가 “쪼개서 주지 마”라고 하면 관련 작업은 가능한 한 한 프롬프트로 묶는다. 단, 진단 결과 없이는 방향을 정할 수 없는 경우만 분리한다.

### 프롬프트 작성 규칙

- 모든 프롬프트, 명령어, 코드는 마크다운 코드블록으로 감싼다.
- 앱 라우트는 반드시 backtick으로 감싼다.
  - 예: `` `/list` ``, `` `/activities/new` ``
  - 이유: Claude Code가 `/list`를 슬래시 명령어로 오인할 수 있음
- 작업 프롬프트에는 항상 다음을 포함한다.
  - 작업 목적
  - 변경 범위
  - 기능/데이터 변경 금지 여부
  - 디자인 토큰 참조 방식
  - 검증 기준
  - commit + push 지시
- 디자인 갈림길에서는 선택지를 제시하고, 필요하면 디자이너 관점의 최적안을 분명히 권고한다.

### 표준 프롬프트 형식

```md
[작업 제목]. 스타일만, 기능/데이터/레이아웃 변경 금지(해당 시). 앱 라우트는 backtick 표기.
스타일 토큰은 `.claude/skills/today-date-design/SKILL.md`에서 읽어 적용. 최소 수정.

[배경]
...

[작업]
...

[중요]
...

[검증]
- `npx next build` 또는 `npm run build` 통과
- 모바일/데스크탑 확인
- 라이트/다크 확인
- 필요한 경우 computed value·스크린샷·픽셀 샘플링 확인

작업 후 `[브랜치명]` 브랜치에 commit + push.
커밋: "design: ..."
```

### 새 브랜치 시작 표준 프롬프트

```md
새 브랜치 `[브랜치명]`를 main에서 생성·체크아웃.
`.claude/skills/today-date-design/SKILL.md`가 보이는지 확인하고, 없으면 main을 merge해 끌어올 것.
준비되면 한 줄 보고.
```

### 머지 표준 프롬프트 (CLI squash 풀 플로우)

Claude Code가 PR 생성 → squash 머지 → 원격·로컬 정리를 한 번에 처리한다. squash 명시로 배포 목록 통제 의도는 유지한다(fast-forward·일반 merge 금지).

```md
`[브랜치명]` 브랜치를 main에 머지하고 정리까지 한 번에 처리해줘.

1. 브랜치가 모두 commit·push 됐는지 확인
2. main 대상으로 PR 생성 (gh pr create)
   - PR 제목: "[타입]: [요약]"
   - PR 본문에 변경 요약 간단히
3. 반드시 squash 방식으로 머지 (gh pr merge --squash --delete-branch)
   - 일반 merge·fast-forward 금지, squash만
4. 머지 후 로컬 main checkout → pull → 로컬 브랜치 삭제

보고: PR URL / 머지 방식(squash 확인) / 최종 main 반영 여부
```

## 4. Git / 배포 운영 원칙

- 작업 단위별로 브랜치를 만든다.
- 미세조정 중간본은 main에 올리지 않는다.
- 브랜치 preview로 확인한다.
- 한 화면/기능/리팩토링 덩어리가 승인되면 main에 머지한다.
- main push 후 Vercel production 반영을 확인한다.
- 브랜치 성격이 다르면 분리한다.
  - 예: 통계 카드 리팩토링과 앱 전체 타이포 표준화는 분리
- 커밋 컨벤션:
  - `feat:` 기능
  - `fix:` 버그 수정
  - `design:` 화면/스타일
  - `refactor:` 구조 개선
  - `docs:` 문서
  - `chore:` 정리
  - `revert:` 되돌리기

### 진단 우선 원칙

원인이 명확하지 않으면 바로 수정 프롬프트를 주지 않는다.

먼저 다음 중 하나를 시킨다.

- 코드로 진단만, 수정 금지
- 정상 케이스 vs 문제 케이스 비교
- git diff로 회귀 추적
- computed value 실측
- 실제 스크린샷 확인
- 같은 역할 요소의 기존 표준·토큰 감사

특히 픽셀, hover, focus, 색상, 높이, 토큰 문제는 눈대중으로 단정하지 않는다.

#### 회색 텍스트 토큰 위계 (260630 v2 확정)

| 토큰 | 라이트 hex | 흰 카드 대비 | 용도 |
|---|---|---|---|
| `--s-ink` | #1a1033 | ≈18:1 | 본문·실제값 |
| `--s-sub` | #6b7280 | 4.83:1 (AA 통과) | 필드명·섹션 라벨·네비·설명, 활성 대기 컨트롤 기본색 |
| `--s-faint` | #9ca3af | 2.54:1 (AA 미달, 의도 수용) | 빈 상태 안내·메타 캡션·빈 별·placeholder |

- 위계: ink(본문값) > sub(라벨) > faint(빈값/메타). 단계 간 명도차 충분(건강).
- **`--s-sub`/`--s-faint`는 분리 유지 필수.** `--s-sub`는 "읽어야 하는 정보 + 활성 대기 컨트롤", `--s-faint`는 "없어도 되는 정보 + 시스템 상태 신호"로 성격이 명확히 갈리고 겹치는 회색 구간 없음. 합치면 상세 화면 라벨↔빈값 구분이 소멸한다.
- `--s-sub` 라이트 4.83:1은 이미 AA 통과(변경 불필요). `--s-faint` 2.54:1 약함은 "안 읽혀도 되는 보조 정보"라 의도된 위계로 수용(진하게 하면 라벨과 안 갈림).

### 라이트/다크 처리 방식

- 앱 전체는 `@media (prefers-color-scheme: dark)` + `--s-*` 토큰 방식으로 다크 처리한다.
- shadcn `.dark` 클래스에 의존하지 않는다.
- Tailwind `dark:` 변형은 이 앱에서 대부분 죽은 규칙이 될 수 있다.
- portal로 렌더되는 요소는 `.page` 토큰을 상속받지 못하므로 `:root` 전역 토큰 또는 portal 자체 토큰 재선언이 필요하다.

### 라이트 카드 원칙

- 라이트모드 컨트롤: 흰 면 + 연한 보더
- 라이트모드 카드: 흰색 + 부드러운 그림자
- 라이트모드에 보라/핑크 배경색을 넓게 깔지 않는다.
- 통일감은 색보다 형태, 그림자, 토큰으로 만든다.

### 그림자 위계

- 추천 카드: 상대적으로 진한 그림자
- 리스트 카드: 중간 그림자
- 통계 카드: 가장 약한 그림자 `--s-card-shadow-xs`
- 조연이라고 그림자를 없애지 않는다. 시스템 안에서 가장 약한 단계로 표현한다.

### 카드/아이콘 원칙

- 카드: `rounded-2xl` 이상, 좌측 정렬
- 카테고리 뱃지: 배경/테두리 없이 아이콘 + 텍스트
- 카드 메타: 칩이 아니라 아이콘 + 흐린 텍스트
- 시스템 이모지 전면 금지, DB 값은 건드리지 않고 표시용 매핑만 사용

## 10. 인증 화면 규칙

### 패스코드

- PasscodeInput은 setup/reset/lock 모두 6자리 고정
- 6자리 입력 전 `onComplete`가 발화하지 않음
- API 검증도 `min(6).max(6)`으로 정합화됨
- reset/setup 안내 문구도 “6자리”로 통일
- 더 이상 “4~6자리”라고 쓰면 안 됨

### 로딩 / fallback

- `setup`·`reset`은 `useSearchParams()` 때문에 Suspense가 필요하다.
- `fallback={null}` 금지. fresh load에서 흰 화면이 생긴다.
- fallback은 `AuthLayout subtitle="..."` 껍데기를 사용한다.
- setup fallback subtitle: “둘만의 공간을 만들어요”
- reset fallback subtitle: “새로운 6자리 패스코드를 설정하세요”

### 높이 정책

- 인증 화면은 `100dvh` 사용
- 일반 화면의 `svh` 정책을 인증 화면에 적용하지 않는다.
- 인증 콘텐츠 높이 문제는 요소를 깎는 방식이 아니라 정렬(`margin:auto`) + 자연 스크롤로 해결한다.

미방문 항목:

- 정보 수정

방문완료 항목:

- 정보 수정
- 기록 수정
- 가보고 싶은 곳으로
- 삭제

- 편집: `size="lg"`, 28~32px

### 날짜

- 저장값: ISO `YYYY-MM-DD`
- 표시: `YYYY.MM.DD (요일)`
  - 예: `2026.05.28 (목)`
- 입력: 네이티브 `<input type="date">`
- 박스 외형은 앱이 통제하고, 달력 팝업만 OS 기본을 수용한다.

- `time_of_day` 값은 `day` / `night` / `any`
- `any`도 숨기지 않고 “아무때나”로 표시
- 아이콘 매핑은 `src/lib/labels.ts`에서 관리
  - `day=Sun`
  - `night=Moon`
  - `any=아이콘 없음`

### 식사시간 칩 (260630 확정)

`/places/[id]`의 식사시간 행에 쓰는 `.mealBadge`는 **정보 표시용 뱃지**다. 카테고리 칩(`.chip`, 36px height 컨트롤)과 역할이 다르므로 height 고정을 주지 않는다.

- font-size: `var(--type-caption)`(12px) 토큰 참조
- padding: 4px·10px **직접 지정**(spacing 전용 토큰 부재로 인한 의도된 예외)
- line-height: 1.4, border-radius: 9999px
- background: `--s-card-border-strong`(라이트 #eceaf3 / 다크 #3a2f4e). 같은 화면 "다녀온 곳" 뱃지(`.visitedTagVisited`)와 동일 토큰
- **금지**: 카드 위에 얹히는 뱃지 배경에 `--s-track`을 쓰지 말 것. `--s-track`은 세그먼트 "눌린 홈" 전용 색이라 다크에서 카드면(`--s-card-bg`)보다 어두워 묻힌다(라이트에선 값이 같아 안 보였던 문제가 다크에서 표면화됨).

### 참고 링크 (260701 확정)

- 저장값은 사용자 입력 원본 유지(마이그레이션 없음), 표시 텍스트도 입력 원본.
- 여는 href만 `src/lib/url.ts`의 `resolveHref(url)`로 스킴 보정: 스킴 없으면 https:// 부착, http(s)/ftp/mailto/tel은 유지, host:8080 포트 오인 방지.
- 스킴 없는 URL(`naver.com`) 입력 허용. 이를 위해 폼 refine·API 4개 라우트의 `z.string().url()` 검증을 완화하고 `isValidReferenceUrl(url)`(폼·API 공용)로 대체.
- 표시: "참고 링크 열기" 고정 텍스트가 아니라 실제 URL + 한 줄 말줄임(`min-w-0 truncate`). 아이콘 external-link(새 탭 열림 의미).
- 링크 열기 동작 `target="_blank" rel="noopener noreferrer"` 유지.

## 12. 리스트 / 카드 규칙

### 카드 내부 수직 간격

카드는 두 덩어리로 읽히게 한다.

1. 헤더 덩어리: 카테고리 + 제목
2. 정보 덩어리: 메타 + 메모

간격 원칙:

- 카테고리 → 제목: 2px 수준
- 제목 → 메타: 12px 수준
- 메타 → 메모: 4px 수준
- 정보끼리는 붙이고, 덩어리 사이는 확실히 띄운다.
- ActivityCard와 PlaceCard는 항상 동일 리듬을 유지한다.

### 시간대 표시

- `time_of_day`는 활동 중심 속성이지만, 값이 있는 곳에서는 공용 라벨 규칙을 따른다.
- 장소에서는 무리하게 동일 필드로 맞추지 않는다.

### 리스트 로딩

- `/list`는 CardGridSkeleton 유지가 현재 결정이다.
- 이유: 카드 한 장의 모양이 예측 가능하므로 스켈레톤이 점프 없이 작동한다.
- `/list` 서버 prefetch는 선택 백로그이며 우선순위 낮음.

## 13. 추천 화면 규칙

### 추천 알고리즘

- 활동 추천과 장소 추천은 분리된 API를 사용한다.
- 방식: 필터 + 점수 + top-shuffle
- 결과는 랭킹이 아니라 동등 추천 묶음이다.
- “다른 추천 보기”는 후보 풀이 반환 개수보다 많을 때만 노출한다.

### 활동 추천

- 시간대
- 소요시간
- 카테고리
- 시간대 매칭:
  - 주간(day)=day+any
  - 야간(night)=night+any
  - 아무때나(any)=전부
- duration:
  - 기본 strict 필터
  - “더 짧은 일정 포함” 옵션
  - “1박 이상” 선택 시 time_of_day 단계 skip

### 장소 추천

- 위치
- 식사 타이밍
- 카테고리
- location은 자유 입력 텍스트 + 부분포함 필터
- 미입력 시 location 필터는 꺼짐

### 입력 단계

- 단일 선택 단계: 선택 즉시 자동 진행
- 카테고리 다중 선택 단계: “추천 받기” 버튼 유지

### 결과 화면 색 위계

- 단색 채움은 “지금 누를 단 하나의 CTA”에만 사용
- 켜짐/선택됨 같은 상태는 틴트 배경 + accent 보더 + accent 글자/체크
- 0개 빈 상태는 활동/장소 공통 패턴으로 유지한다.
- 0개 상태의 추가 버튼은 단독 진입 버튼으로 그라데이션 허용

### 결과 하단 버튼

결과가 1개 이상일 때:

1. 더 짧은 일정 토글
2. 다른 추천 보기
3. 처음부터

0개 상태:

- 박스 아래 “다른 추천 보기” 없음
- “처음부터”만 유지

## 14. 다이얼로그 규칙

### 닫기 수단

- 입력 폼 다이얼로그: X 버튼만 유지, 하단 취소 제거
- 결정 다이얼로그: 취소 버튼만 유지, X 제거
- 같은 의미의 닫기 액션을 중복하지 않는다.

### 제목 위계

- 다이얼로그 제목은 페이지 h1 토큰과 분리한다.
- 전용 사이즈: `text-xl` 20px / `font-semibold`
- 좁은 다이얼로그에 페이지 h1(26~28px)을 강제하지 않는다.

### 다크 대응

- 다이얼로그는 portal로 `.page` 밖에 렌더된다.
- `.page` 토큰 상속을 기대하지 않는다.
- `@media(prefers-color-scheme:dark)` + `--s-*` 재선언 또는 전역 토큰을 사용한다.
- 오버레이는 다크에서 더 진하게, 팝업은 ring/shadow로 분리한다.

대상:

- `/places/[id]`

### 홈 통계 prefetch

홈 통계도 클라이언트 fetch 때문에 막대 후 숫자 공백이 보였으므로 서버 prefetch + `initialData`로 정리했다.

관련 공용 함수:

- `src/lib/data/activities.ts`
- `src/lib/data/places.ts`
- `src/lib/data/dashboard.ts`

### 프로그래매틱 전환

Link 전환은 top loader가 잡지만, `router.push()` 계열 프로그래매틱 전환은 별도 처리해야 한다.

- `useTopLoader().start()`를 fetch 직전에 호출
- 적용 예:
  - `lock/page.tsx` unlock
  - `setup/page.tsx` passcode
  - `reset/page.tsx` reset
  - `HomeMenu.tsx` logout
- 성공 경로에서는 `done()`을 수동 호출하지 않는다.
  - pushState 자동 done에 맡겨야 막대 끝 = 화면 등장 싱크가 맞는다.
- 에러/네트워크 실패 경로에서만 `done()` 명시

### setup/reset fallback

- `fallback={null}` 금지
- AuthLayout fallback으로 브랜드/배경을 먼저 보여준다.

- `--s-row-*`, `--s-chip-*`, `--s-check-*` 전용 토큰 23개 제거

### 배경

- computed value가 맞아도 실제 눈에 안 보일 수 있으므로 스크린샷·픽셀 샘플링 확인이 필요하다.

### 홈 레이아웃 확정값 (260625)

타이포 표준에서 통계 제목을 15px로 키운 뒤 빡빡함이 생겨 여러 시도 끝에 확정했다.

- **추천 CTA 카드**: padding 20px, 아이콘↔텍스트 16px. 주연 카드이므로 줄이지 않는다.
- **통계 행**: 상하 padding 12px, 아이콘↔텍스트 12px. 15px 제목 무게에 맞춘 값.
- **홈 폭**: 모바일 `max-w-xl`(576px) / PC `max-w-3xl`(768px).
- **정렬**: 추천·통계 모두 좌측. 가운데 정렬은 검토 후 기각(3요소 세로 스택이라 좌측 라인 정돈이 더 깔끔, 화면 내 정렬 기준 일관성).
- **위계**: 페이지 h1(주연) > 추천 카드(준주연, 16px) > 통계 행(조연, 15px). 추천 제목 16px은 리스트 카드 18px과 구분되는 자리라 키우지 않는다.

#### 시행착오 교훈 (재시도 금지)

- PC 홈 폭을 896px로 넓히고 모바일 추천 카드 padding을 줄여 통계에 공간을 재배분하려 했으나 역효과였다: PC에서 추천 2열이 과하게 벌어지고 통계 행이 횡으로 늘어졌으며, 모바일에서 추천 카드가 주연 존재감을 잃었다. → 원복하고 통계 여백 개선만 유지.
- **홈 폭은 화면별 개별 선언**(공통 컨테이너 없음): 홈만 바꿔도 `/list`(896px)·상세(512px)에 영향 없으나, 폭 확장 자체가 홈에 역효과라 768px 유지가 맞다.
- PC 홈 상단 여백 과다는 폭이 아니라 콘텐츠 배치로 풀어야 한다(백로그).

## 17. 타이포그래피 표준 (260625 완료·main 반영)

260623 착수, 260625에 `feat/typography-scale`로 앱 전체 타이포 스케일 표준화를 완료해 main에 반영했다.

### 문제 (배경)

콘텐츠 텍스트 크기×weight 조합이 18종 이상으로 과다했다. 카드 주제목이 리스트 18px / CTA 16px / 통계 13px로 제각각이었고, 페이지 h1도 상세 27px / 위저드 20px로 충돌했으며, 구분 라벨·카드 제목에 공통 표준 클래스/토큰이 없었다.

### 7개 역할 토큰 구조 + 유틸 클래스

| 토큰 | 모바일 | PC | weight | color | 유틸 클래스 | 수렴 대상 |
|---|---:|---:|---:|---|---|---|
| `--type-page-title` | 27 | 30 | 600 | `--s-ink` | `.pageTitle` | 페이지 h1(상세·폼) |
| `--type-card-title` | 18 | 18 | 600 | `--s-ink` | `.cardTitle` | 리스트·인증 카드 |
| `--type-card-title-sm` | 15 | 16 | 600 | `--s-ink` | `.cardTitleSm` | 홈 통계 행·CTA |
| `--type-section-label` | 12 | 12 | 600 | `--s-sub` | `.sectionLabel` | 섹션 헤더(독립 구획) |
| `--type-body` | 14 | 14 | 400 | `--s-ink` | `.bodyText` | 본문·값 |
| `--type-preview-meta` | 12 | 12 | 400 | `--s-label-muted` | `.previewMetaText` | 미리보기·메타 |
| `--type-caption` | 12 | 12 | 400 | `--s-faint` | `.captionText` | 캡션·메타(등록일 등) |

### 사용 규칙

- 새 화면 텍스트는 위 7역할 중 하나로 분류해 해당 유틸 클래스를 쓴다. 임의 size/weight/color 직접 지정 금지.
- 색 토큰(`--s-sub`·`--s-faint` 포함)은 `:root` 전역 정의라 portal·인증 화면도 상속 가능하다(타이포 작업 중 승격 완료).

### 라벨 위계 — "라벨"은 한 종류가 아니다 (중요)

같은 라벨처럼 보여도 정보 구조상 계위가 다르므로 두 역할로 분리한다.

- `.sectionLabel`(600): **섹션 헤더**. 위/아래 콘텐츠 블록을 나누는 독립 구획 제목. 값에 종속되지 않고 홀로 떠 또렷해야 함. 예: 홈/리스트 "가보고 싶은 곳"·FilterGroup 라벨.
- `.fieldLabel`(500): **필드명**. 바로 아래 값과 1:1로 붙어 그 값을 설명. 값에 종속되므로 값보다 약해야 함. 예: 상세 "소요시간"·"위치"·"메모". size·color는 section-label 공유(12/`--s-sub`), weight만 500.
- 판정 기준: 독립적으로 구획을 나누면 600(sectionLabel), 특정 값에 붙은 이름이면 500(fieldLabel).

### 상세 화면 값 위계

실제값(`--s-ink`) > 필드명 라벨(`--s-sub`) > 빈값(`--s-faint`).

- 빈값("아직 메모가 없어요")은 `--s-faint` 유지. 라벨을 sub로 올리면 빈값과 자동 분리됨.
- 등록일 등 메타 캡션은 `.captionText`(`--s-faint`) 유지(가장 약한 보조 정보).

### 손대지 않은 것 — 의도된 예외 (7토큰 비대상)

- 위저드 h1 20px: 키우면 레이아웃 흔들림, 미반영 화면 스코프.
- 다이얼로그 제목 20px: 페이지 h1과 분리된 전용 사이즈.
- 카드 메모 미리보기·빈 상태 14px/`--s-faint`(ActivityCard·PlaceCard·VisitRecordBlock): body도 caption도 아닌 "카드 흐린 미리보기" 의도된 조합.
- DetailBlock h2 블록 제목 14px/500: 필드 라벨 톤, 별도 위계.
- 인증(`/lock`, `/setup`) 팔레트 일부, 폼 입력, 버튼, 탭 컨트롤.

### 토큰만 빌려 쓰는 패턴 (260630 추가)

`--type-caption`(12px) 토큰은 캡션뿐 아니라 `.mealBadge` font-size에도 참조된다. 단 유틸 클래스 `.captionText`를 통째로 적용하면 color(`--s-faint`)까지 끌려오므로, 뱃지 자체 색을 유지하려면 **font-size 토큰만** 참조한다. 색이 다른 요소에 타이포를 맞출 때는 유틸 클래스 통째 적용이 아니라 필요한 토큰만 빌려 쓴다.

### 작업 교훈

- **토큰 의도를 역으로 읽지 말 것**: 통계 행 제목이 "현재값(13/500)이 토큰 정의(15/600)와 다르다"는 이유로 수렴 후보에서 누락됐었다. 그러나 `--type-card-title-sm`은 애초에 13→15 수렴을 위해 만든 토큰이었다. 값이 다른 게 정상이고 그 차이를 없애는 게 토큰의 목적 — "값이 달라서 제외"가 아니라 "달라서 적용 대상"이다.
- **`:root` 승격 시 픽셀 불변 전수 검증**: `--s-sub`·`--s-faint` 승격 시 fallback 값과 승격값이 일치해 픽셀 0이었으나, 어긋났다면 라이트 색이 틀어졌을 것. 승격은 경로별 검증 필수.
- **공용 클래스 통째 변경 금지**: `.captionText`에 필드 라벨과 진짜 메타 캡션이 섞여 있어, 라벨만 올리려면 별도 클래스(`.fieldLabel`) 분리가 필요했다. 같은 클래스를 쓰는 다른 용처 영향을 먼저 감사한다.
- **토큰 통합/분리는 사용처 성격으로 판단한다 (260630 v2)**: `--s-hint`는 홈 전용 2곳·값이 의미와 역행하는 죽은 토큰이라 `--s-faint`로 통합했다. 반면 `--s-sub`/`--s-faint`는 둘 다 살아있고 성격이 명확히 갈려(읽어야 함 vs 없어도 됨) 분리 유지가 맞았다. "통합이 항상 정리"가 아니라, 사용처 성격이 겹치는지로 가른다. 레이아웃이 이미 위계를 만들고 있으면(라벨 아래 빈값 배치) 색으로 한 층 더 가를 필요가 없다.
- **진단 수치도 재검증 대상이다 (260630 v2)**: `--s-sub` 라이트 대비를 4.35:1(AA 미달)로 보고했던 게 채널 역감마 계산 오류였고, 정확한 값은 4.83:1(AA 통과)이었다. 한 번의 재계산으로 "고쳐야 할 빚"이 "고칠 것 없음"으로 뒤집혔다. 접근성·대비 수치가 결론을 좌우할 때는 계산 과정을 재확인한다.
- **사용자 관찰과 어시스턴트 추정을 구분한다 (260630 v2)**: "빈값이 약하다"는 사용자가 관찰한 적 없고 이전 어시스턴트가 추정해 백로그에 넣은 것이었다. 추정을 사용자 신호로 잘못 귀속하면 불필요한 전역 작업으로 번진다. "약하다/흐리다"의 출처를 확인하고, 추정이면 객관 기준(접근성 수치 등)으로만 평가하며, 의도된 위계면 닫는다.

## 20. 알려진 교훈 / 금지 사항

- 추측으로 픽셀·색·높이를 단정하지 않는다.
- computed value만 보고 끝내지 않는다. 실제 스크린샷과 사람 눈 검증이 필요하다.
- 필요하면 실제 렌더 픽셀 색을 샘플링해 명도차를 본다.
- 사용자 관찰은 높은 신뢰도로 재검토한다.
- 확정 규칙처럼 보이는 것도 사용자 합의 없이 문서에 박힌 경우가 있을 수 있다.
- 여러 번 수정해도 안 잡히면 즉시 진단 모드로 전환한다.
- 라이트 렌더를 바꾸는 토큰 정리는 픽셀 불변 여부를 반드시 확인한다.
- portal 요소는 `.page` 토큰 상속을 받지 못한다.
- 스켈레톤은 모든 로딩의 정답이 아니다.
- 새 규칙은 문서부터 박지 말고 실물 코드로 확인 후 문서화한다.
- “기존 표준부터 검증”이 즉흥 hex와 임의 위계를 막는 핵심 안전장치다.
- 위계는 없음/있음이 아니라 강도 단계로 표현한다.
- 다크모드는 명도 방향이 라이트와 반대다. 다크에서 강조/약화가 역전되지 않는지 별도 확인한다.
- 표준 수립은 전면 재설계가 아니라 기존 값의 역할별 수렴으로 접근한다.
- UI가 막아준다고 구조적으로 안전한 것은 아니다. API 검증도 함께 맞춰야 한다.
- **토큰의 목표값과 현재값이 다른 건 정상이다.** 그 차이를 없애는 게 토큰의 존재 이유이므로, "값이 달라서 수렴 제외"로 역해석하지 않는다.
- **같은 역할로 보여도 계위가 다르면 분리한다.** "라벨" 하나로 묶지 말고 독립 구획 헤더(sectionLabel)와 값에 종속된 필드명(fieldLabel)을 구분한다. 표준화는 무조건 한 클래스로 합치는 게 아니라 역할을 정확히 가르는 것이다.
- **공용 클래스를 통째로 바꾸기 전에 모든 용처를 감사한다.** 한 클래스에 다른 역할이 섞여 있으면 일부만 분리해야 한다.
- **레이아웃 재배분은 한 화면만 보고 결정하지 않는다.** 폭·여백은 다른 화면 영향과 전체 비율을 함께 본다. 한쪽을 살리려 다른 쪽을 깎는 트레이드는 대개 역효과다(추천 축소로 통계 살리기 실패 사례).
- **멀쩡한 것을 더 건드리면 깨진다.** 만족스러운 상태에 도달하면 멈춘다(홈 추천 제목 16px, PC 폭 768px 등).
- **머지는 CLI squash 풀 플로우다.** Claude Code가 PR 생성 → `gh pr merge --squash --delete-branch` → 로컬 main pull·브랜치 삭제를 한 번에 처리한다. squash 명시로 배포 목록 통제(fast-forward·일반 merge 금지).
- **백로그 항목은 실측 전까지 "진짜 빚"으로 단정하지 않는다.** 이전 어시스턴트가 텍스트로만 남긴 stale 항목이 섞여 있다(260630에 입력 step·버튼 max-width 2건이 "코드는 이미 멀쩡"으로 닫힘). 작업 진입 전 진단 프롬프트로 실측 확인 먼저.
- **진단 시 컴포넌트의 렌더 분기를 혼동하지 않는다.** 한 컴포넌트가 조건부로 다른 컨테이너를 쓰면(마법사 화면 512px vs 결과 화면 896px) 어느 분기의 값인지 확인해야 엉뚱한 후보를 잡지 않는다(260630 버튼 max-width 1차 오독 사례).
- **`--s-track`은 세그먼트 "눌린 홈" 전용 색이다.** 카드 위에 얹히는 뱃지/요소 배경에 재사용하면 다크에서 카드면보다 어두워 가라앉는다. 카드 위 정보 뱃지 면색은 `--s-card-border-strong`. 라이트에서 두 토큰 값이 같아도 다크에서 갈라지므로 다크 별도 확인 필수.
- **화면 성격이 다르면 같은 요소라도 폭/크기가 달라도 된다.** 버튼 폭이 질문 512px / 결과 320px / 갤러리 896px로 다른 건 불일치가 아니라 각 화면에 최적화된 의도. 통일이 항상 정답은 아니다.
- **진단이 내놓은 수치도 결론을 뒤집을 만하면 재검증한다.** `--s-sub` 대비를 4.35:1(AA 미달)로 보고했던 게 계산 오류였고 실제 4.83:1(통과)이었다. 접근성·대비처럼 수치가 결론을 좌우하면 계산 과정을 다시 확인한다. "백로그는 실측 전까지 진짜 빚으로 단정 말 것"에 더해, 그 실측 수치 자체도 의심 대상이다.
- **사용자 관찰 ≠ 어시스턴트 추정.** "약하다/흐리다"가 사용자 신호인지 이전 어시스턴트가 추정해 남긴 것인지 출처를 확인한다. 추정을 사용자 신호로 잘못 귀속하면 의도된 위계를 깨는 불필요한 전역 작업으로 번진다. 추정이면 객관 기준으로만 평가하고, 의도된 설계면 근거 기록 후 닫는다.
- **토큰 통합은 사용처 성격이 겹칠 때만.** `--s-hint`(죽은 토큰)는 통합, `--s-sub`/`--s-faint`(성격이 갈리는 살아있는 토큰)는 분리 유지. 합치면 위계가 소멸하는 토큰은 절대 통합하지 않는다.
- **요소에 직접 background를 칠하는 패턴은 `min-height: 100svh`를 동반해야 한다.** 없으면 짧은 콘텐츠에서 배경이 콘텐츠 높이까지만 칠해지고 그 아래가 노출된다(그라데이션 전역 확장 시 `.pageStatic min-height: auto` 버그).
- **회귀 여부는 "언제 보였나"가 아니라 "어디서 비롯됐나"로 판단한다.** 추가 플로우 조건 미유지는 등록 후 상세 이동(PR #32) 이후 표면화됐으나, 진단 결과 뿌리는 리스트→추가 진입부 미배선(기존 동작)이었다. 표면화 시점으로 회귀를 오판하지 않는다.
- **입력을 막는 검증을 먼저 본다.** 링크 스킴 보정은 표시(href)만 고쳐선 안 되고, 애초에 스킴 없는 입력을 차단하던 폼·API zod 검증을 함께 완화해야 실제 동작한다. 표시만 보지 말고 입력 파이프라인 전체를 확인한다.
